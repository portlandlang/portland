# 0013 ŌĆö The `case/in` spec: exhaustive, static, Ruby-shaped

- **Status:** Accepted (the pattern-edge calls in ┬¦4 are **Tentative** ŌĆö
  agent-recommended and not yet felt by users; revisit cheaply with usage)
- **Date:** 2026-07-22
- **Issue:** [#20](https://github.com/portlandlang/portland/issues/20)

Pattern matching is load-bearing (ADRs 0005/0008/0009 lean on `in nil` /
`in some(nil)` / struct patterns). Both of Ruby's forms survive, with
Ruby's own split kept: **`case/when`** compares values, **`case/in`**
matches shapes and binds. Five decisions:

## 1. Exhaustiveness is compile-checked

A `case/in` must cover its subject or say `else`. A maybe missing
`in nil` refuses to build, naming the missing case. This is the other
half of the optionals feature ŌĆö the toolkit *offers* handling; `case/in`
is where handling is *proven complete*. Divergence from Ruby's runtime
`NoMatchingPatternError` is the loud kind (compile error), per the
migration promise. A user note from this decision's session: pattern
matching must be learnable from Portland's own docs ŌĆö Ruby familiarity
is not assumed (the deciding user had never used Ruby's).

## 2. `when` runs on statically resolved `===`

`===` is an ordinary method, resolved at compile time: types may define
it; the fallback is `==`. `when Integer`, `when 1..9`, `when /re/`, and
user types all keep working; only runtime redefinition dies (already
cut-list). Pragmatics: `when` over arbitrary `===` can't always be
proven complete, so non-exhaustive `case/when` requires `else`;
`case/in`'s structural patterns carry the real completeness proof.

## 3. Captures stay Ruby's ŌĆö fenced

A bare lowercase name in a pattern is a capture, exactly as in Ruby
(polyfill tier-1 compatibility). The `in none`-style silent-catch-all
trap is neutralized by rules Portland has and Ruby doesn't:

- **No-shadow:** a capture colliding with any existing local or method
  is a compile error.
- **Exhaustiveness:** a catch-all makes later arms unreachable ŌĆö
  unreachable arms are compile errors.
- **Unused-capture lint:** bound-but-never-read captures are flagged.

## 4. The pattern edges (Tentative)

- **Pin `^variable`** ŌĆö in. The compare-don't-capture escape hatch that
  makes ┬¦3 livable.
- **Guards `in pattern if condition`** ŌĆö in. Guarded arms don't count
  toward exhaustiveness (the compiler can't prove an `if`); unguarded
  coverage is still required.
- **Alternatives `in 1 | 2`** ŌĆö in. `when`'s comma, pattern-flavored.
- **One-line forms** ŌĆö in, both: `expr => pattern` (destructure, loud on
  mismatch) and `expr in pattern` (boolean test). `=>` also resolves the
  destructuring question ADR 0011 left unentangled: `a, b = pair` stays
  out; `pair => [a, b]` provides the capability inside pattern grammar.
- **Find pattern `in [*, x, *]`** ŌĆö out for now, demand-driven; returns
  when a real file pulls for it.

## 5. Struct patterns ŌĆö in, keyword-only

`in ReturnNode(value: nil)` matches by type and field; `in Node(field:)`
binds. Structs are born deconstructable ŌĆö fields are the pattern
surface, no `deconstruct_keys` ceremony. Positional struct patterns are
**out**: keyword-only, matching `new`, for the same order-decoupling
reason as ADR 0011.

## Consequences

- The trio's `case node.kind when "return"` dispatch can become real
  pattern matching once this is built (successor build issue to #20).
- The nested-maybe story completes: `in nil` vs `in some(nil)` in
  patterns, exhaustively checked.
- The seed builds the runtime half (matching, binding, pin, guards,
  alternatives, one-line forms, struct patterns); exhaustiveness,
  no-shadow-in-patterns, unreachable-arm errors, and the unused lint are
  the static half (#9), previewed as runtime panics where feasible.
- Ledger: `docs/ruby/pattern-matching.md`.
