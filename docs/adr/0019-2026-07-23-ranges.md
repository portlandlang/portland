# 0019 — Ranges: total coverage, slices are collections, never-guess ends

- **Status:** Accepted (not yet built)
- **Date:** 2026-07-23

## Context

Ranges were parked as "three small questions." Two turned out to have
real substance; one answered itself.

The seed has no range support at all today, so this is greenfield. Ruby's
range behavior was verified against Ruby 4.0.6 rather than recalled —
several of its edges are sharper than reputation suggests.

## Decision

Ranges are **first-class values**: `span = 1..5`, passed around,
iterated. `..` is inclusive, `...` exclusive — Ruby's spellings.

### 1. Exhaustiveness proves contiguous integer coverage

Range patterns are in (`in 1..5`), and ADR 0013's compile-checked
exhaustiveness **understands them**: sort the integer ranges by start,
require a beginless first and an endless last, require no gaps between
neighbors. That proves this total, with no `else`:

```ruby
case score
in ..0    then "none"
in 1..9   then "some"
in 10..   then "lots"
end
```

Anything the checker can't prove falls back to ADR 0013's existing
rule — unprovable means an `else` is required. Interval reasoning gets
genuinely hard once ranges mix with guards and other pattern kinds; that
fallback is where the hard cases go, permanently.

The alternative — ranges never count toward exhaustiveness, so range
cases always need an `else` — was rejected once its real cost surfaced.
An empty `else` is not a no-op: per ADR 0012 it yields `nil`, infecting
the result type with an absence that can never occur and taxing every
caller through the ADR 0008 toolkit. The only type-clean filler is
`else panic "unreachable"`, which dilutes the thing ADR 0010 sold `panic`
on — that `grep panic` is a crash audit of *accepted risks*, not a list
of places the compiler couldn't keep up. Forcing authors to write
unreachable panics would corrode a load-bearing guarantee to save
analysis that isn't hard in the shape that matters.

Neither option was cheaper to build: the static exhaustiveness checker
does not exist in any form yet (#9 owns it; the seed previews it as a
runtime no-match panic). With build cost equal, specify the good one.

**Gaps are errors; overlap is legal; order is not the compiler's
business.** Overlap must stay legal because first-match-wins is real
semantics and the cascading idiom depends on it:

```ruby
case size
in ..10   then "small"      # every arm overlaps every earlier one
in ..100  then "medium"
in ..1000 then "large"
end
```

The subset of overlap that *is* a bug — an arm entirely covered by
earlier arms — is already an error under ADR 0013 §3 (unreachable arms),
which generalizes to ranges for free. Arm *order* carries no meaning
when arms are disjoint (the checker sorts internally to prove coverage),
so it is style, not semantics: a lint that suggests, never a format that
rewrites, since safely reordering requires the disjointness proof first.
No Portland formatter exists or is planned; this waits for one.

### 2. A slice is always a collection, never a maybe

`array[1..3]` returns `Array`. `string[1..3]` returns `String`. A start
outside the collection clamps, yielding empty:

```ruby
words[1..3]      # Array — no unwrap, ever
words[99..]      # []
"hello"[9..]     # ""
```

This diverges from Ruby, whose actual rule (verified) is asymmetric —
the end clamps freely but the start does not:

| expression | Ruby | Portland |
|---|---|---|
| `[1,2,3][1..99]` | `[2,3]` | `[2,3]` |
| `[1,2,3][3..]` | `[]` | `[]` |
| `[1,2,3][4..]` | `nil` | `[]` |
| `[1,2,3][-99..]` | `nil` | `[1,2,3]` |
| `[1,2,3][2..1]` | `[]` | `[]` |

**ADR 0010 is untouched: `array[1]` still returns `T?`.** The two are
different questions. Asking for *one element* has an honest absence
answer — it is there or it is not. Asking for a *sub-collection* does
not: a sub-collection of nothing is `[]`, a perfectly good collection.
Ruby's `[4..] → nil` is the nil/empty conflation Portland exists to
kill, and Ruby's own end-clamping shows the instinct was already there —
it just stopped halfway.

The ergonomics decide it too: making slices maybes taxes *every* slice
with `or []` to guard an edge case that is rare.

The divergence stays loud where it counts. Code that checks
`slice.nil?` hits ADR 0007's never-absent-left-side error; `if
array[4..]` is already an error (no truthiness); and code that does not
check would have *crashed* in Ruby on `nil.each`. Loud where checked,
Ruby-crashes-→-Portland-works where not — no silent wrong answers.

**Open to revision on evidence.** "Almost no real code relies on
nil-from-slice" is an assumption, not a measurement. It is a good
ruby_research query once the full corpus lands; this decision is a prior
to affirm or overturn, not a closed book.

### 3. Endless and beginless, with a never-guess boundary

Both are in — decision 1 requires them, since bounded ranges cannot
cover integers' infinite tails — and they work everywhere ranges work
(values, slices, patterns).

A range **closes when the next meaningful token cannot continue an
expression** (`end`, `then`, `else`, `in`, `when`, `]`, `)`, `}`, `,`,
end of file). That makes real usage paren-free:

```ruby
array[1..]                 # ] closes it
slice(1.., 2..)            # , and ) close it
in 10.. then "big"         # then closes it
```

Where the next token *could* be an operand, that is a compile error
naming both readings — the ADR 0016 shape, one paren away from
compiling:

```ruby
span = 1..
p span
# error: endless range at end of line — does it continue?
#   the range, closed here:  span = (1..)
#   a range up to p span:    span = 1..p(span)
```

Ruby resolves this by silently continuing across the newline: `x = 1..`
followed by a line containing `5` really is `1..5` in Ruby 4.0.6,
indented or not. That is why "a newline just closes the range" was
rejected — it would reinterpret that code with no error. The narrow spot
where Portland errors is precisely where Ruby is a footgun.

**The mirror rule: a line-initial `..` after a complete expression is
the same error.** A beginless range at the start of a line has two
genuine readings whenever the previous line ended complete:

```ruby
x = compute
..4
# error: `..` at the start of a line — does it continue the line above?
#   a bounded range:          x = compute..4
#   a separate beginless one: x = compute  /  ..4
```

Real beginless usage is unaffected, because it always follows a token
that cannot end an expression — `in ..10`, `array[..5]`, `foo(..5)` are
unambiguous and paren-free.

Ruby's answers here are three unrelated parser accidents rather than a
principle to inherit: trailing `..` reaches *forward* across a newline,
leading `..` does *not* reach back, and leading `.` does. Verified on
4.0.6, and the split survives even inside parentheses — `y = (1` /
`..4)` binds `y` to `..4`, not to `1..4` and not to `1`. Portland
replaces all three accidents with one rule: **a range spans a newline
only where exactly one reading exists; otherwise, name both and ask.**

## Consequences

- Build is a day's work in the seed plus the trio: `..`/`...` tokens, a
  range value, slicing, range patterns, the boundary rule. The
  exhaustiveness half is spec only — #9 builds it, and until then the
  seed's runtime no-match panic previews it, as it does for every other
  pattern today.
- Lexer interaction with ADR 0018: `1..5` must not read `1.` as the
  start of a float, and `...` needs a three-character operator alongside
  `..` and `.`.
- `(1..n).each` becomes the counted-loop idiom, which is the main reason
  ranges are worth having this early.
- Ledger: `docs/ruby/ranges.md`. Migration: range literals, patterns,
  and in-bounds slices compile verbatim; out-of-range slices change nil
  to empty (loud where checked); a trailing endless range in an
  assignment needs its parens.
